TENANT GUARDEntra security & compliance console

Find & clear inactive accounts
across your Entra tenant

Tenant Guard surfaces members and guests with no interactive or non-interactive sign-in beyond your chosen window, and lets an administrator remove them — directly against Microsoft Graph, with no backend.

What this requests (delegated, admin-consented)

  • ●
    User.Read.All & AuditLog.Read.All — read users and their sign-in activity
  • ●
    Directory.Read.All — read tenant license state (required for sign-in activity)
  • ●
    Device.Read.All — read registered devices and their last sign-in
  • ●
    IdentityRiskyUser.Read.All, Policy.Read.All, UserAuthenticationMethod.Read.All, ServiceHealth.Read.All — security views (risky users, Conditional Access, MFA, service health)
  • ●
    BitlockerKey.ReadBasic.All & DeviceManagementManagedDevices.Read.All — compliance & antivirus views: BitLocker recovery-key metadata (no key material), Intune encryption status, Defender antivirus health, and active-malware reporting
  • ●
    User.ReadWrite.All — delete selected user accounts (soft-delete, recoverable 30 days)
  • ●
    Directory.AccessAsUser.All — requested only at the moment you delete a device (Graph has no narrower delegated permission for that operation); permanent, no recovery
  • ●
    Needs an Entra ID P1/P2 license. Deleting users needs User Administrator or Global Administrator; deleting devices needs Cloud Device Administrator, Intune Administrator, or Global Administrator.

Configuration

The default client ID is baked into this page. An organization that prefers its own app registration can paste its own multi-tenant SPA client ID here. "Grant admin consent" sends a Global Administrator through the consent screen, which provisions the enterprise application (service principal) in the tenant automatically — no app needs to be created by hand.
